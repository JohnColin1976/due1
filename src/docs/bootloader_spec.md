Ниже — спецификация требований (SRS) для добавления UART-загрузчика на SAM3X8E для программирования через USART0 без GPIO, с описанием порядка действий со стороны T113 (/dev/ttyS4). Формат сделан так, чтобы его можно было напрямую передать в CODEX как задачу на изменение существующего проекта.

SPEC: UART Bootloader для SAM3X8E по USART0 без GPIO (T113 ttyS4 → SAM3X USART0)
0. Контекст и цель

Цель: реализовать обновление прошивки SAM3X8E через UART канал USART0, где:

T113 (Linux) получает прошивку по Ethernet и передаёт её по /dev/ttyS4

SAM3X8E принимает прошивку в bootloader и программирует Flash

GPIO линии RESET/ERASE не используются

Вход в bootloader выполняется через команду приложению + software reset и флаг в RTC_GPBR

1. Определения

BL — bootloader в Flash SAM3X8E

APP — основное приложение

USART0 — UART интерфейс на SAM3X8E, подключённый к T113 /dev/ttyS4

GPBR — RTC General Purpose Backup Register (RTC_GPBR[x])

MAGIC_UPDATE — значение флага для входа в BL

2. Высокоуровневая архитектура
2.1. Размещение в памяти

BL размещается в начале Flash и запускается первым.

APP размещается после BL.

Требования к адресам (параметризуемые):

FLASH_BASE = 0x00080000

BL_START = FLASH_BASE

BL_SIZE = 32 KB (допускается 16 KB, но целевое 32 KB)

APP_START = BL_START + BL_SIZE

BL обязан программировать только область APP ([APP_START .. FLASH_END]), не затрагивая BL.

2.2. Вход в режим обновления без GPIO

Вход в BL осуществляется через:

APP получает команду “перейти в обновление”

APP записывает MAGIC_UPDATE в RTC->RTC_GPBR[0]

APP вызывает NVIC_SystemReset()

BL при старте читает GPBR и входит в режим обновления

3. Функциональные требования: Bootloader (SAM3X8E)
3.1. Поведение при старте

BL должен:

Инициализировать минимальные системные настройки, необходимые для USART0 и Flash.

Проверить флаг обновления:

flag = RTC->RTC_GPBR[0]

если flag == MAGIC_UPDATE, то:

очистить флаг: RTC->RTC_GPBR[0] = 0

перейти в режим UART-обновления (см. раздел 4)

иначе перейти к “обычной загрузке” (jump to APP) с опциональным коротким окном ожидания SYNC.

Окно ожидания SYNC (опционально, но рекомендуется):

Если GPBR флага нет, BL может ожидать T_SYNC_WAIT_MS = 150 ms входной синхронизации по UART.

Если синхронизация получена — вход в режим обновления.

Если нет — jump to APP.

3.2. Jump to Application

BL должен корректно передавать управление APP:

Установить SCB->VTOR = APP_START

Загрузить MSP из *(uint32_t*)APP_START

Перейти на Reset_Handler APP из *(uint32_t*)(APP_START+4)

Перед прыжком отключить прерывания __disable_irq()

3.3. USART0

BL должен поддерживать USART0:

8N1

без RTS/CTS

скорость по умолчанию: 115200

поддержка 921600 как compile-time option (опционально)

Параметры USART0 должны быть едиными для BL и T113.

3.4. Программирование Flash

BL должен:

выполнять erase и write только в диапазоне APP.

программировать по страницам согласно требованиям SAM3X EFC (EEFC).

ждать завершения операций (FRDY).

возвращать детальные ошибки при таймауте/ошибке EEFC.

запрещать запись по адресам вне допустимого диапазона.

3.5. Проверка целостности

BL должен поддерживать проверку:

CRC32 для всего образа либо для каждого блока (минимум — CRC32 всего образа).

При несовпадении CRC BL должен завершать с ошибкой и НЕ запускать APP автоматически (оставаться в режиме обновления).

4. Протокол обмена T113 ↔ Bootloader (USART0)
4.1. Общие требования

Протокол бинарный, устойчивый к шуму: синхронизация, длина, CRC.

Все числа в протоколе: little-endian.

BL обязан отвечать ACK/NAK на каждую команду.

BL должен поддерживать ограничение размера кадра (например 1024 или 2048 байт payload).

4.2. Синхронизация

SYNC кадр:

Host (T113) отправляет: 0x55 0xAA 0x55 0xAA

BL отвечает: ASCII BL>OK\n (или бинарный ACK)

4.3. Команды (минимальный набор)

Команды протокола:

CMD_INFO (0x01)

Host → BL: запрос

BL → Host: версия BL, APP_START, допустимый диапазон, page size, max chunk

CMD_ERASE (0x02)

Host → BL: addr_start, length

BL → Host: ACK/ERR

Ограничение: addr_start должен быть ≥ APP_START

CMD_WRITE (0x03)

Host → BL: addr, length, crc32(data), далее data[length]

BL → Host: ACK/ERR

BL обязан:

проверять addr диапазон

проверять CRC блока

программировать Flash и подтверждать запись

CMD_VERIFY (0x04)

Host → BL: addr_start, length, crc32_total

BL → Host: ACK/ERR (+ фактический crc при ошибке опционально)

CMD_RUN (0x05)

Host → BL: без параметров

BL → Host: ACK и выполняет jump to APP

CMD_ABORT (0x06) (опционально)

BL прекращает текущую операцию и возвращается к ожиданию команд.

4.4. Формат ответа

Успех: ACK (0x79) + status=0

Ошибка: NAK (0x1F) + error_code + optional_detail

4.5. Коды ошибок (минимум)

0x01 — invalid command

0x02 — invalid address/range

0x03 — invalid length

0x04 — CRC mismatch (block)

0x05 — flash program error

0x06 — flash erase error

0x07 — timeout waiting data

0x08 — verify failed (total CRC)

0x09 — internal state error

5. Требования к APP (SAM3X8E)
5.1. Команда входа в обновление

APP должен поддерживать команду по USART0 (принимаемую от T113):

Триггер “ENTER_BOOT” (например ASCII BOOT\n или бинарная команда в существующем протоколе)

При получении:

RTC->RTC_GPBR[0] = MAGIC_UPDATE

(опционально) flush/ответ T113 “OK, rebooting”

NVIC_SystemReset()

5.2. Совместное использование USART0

APP и BL должны использовать одинаковые параметры USART0 (baud/8N1).

APP не должен оставлять USART0 в состоянии, препятствующем BL (например, не отключать жизненно важные такты, не переводить пины в GPIO без восстановления, если ожидается обновление).

6. Требования к T113 (Linux, /dev/ttyS4): порядок действий обновления
6.1. Предусловия на T113

/dev/ttyS4 доступен и не занят getty/console сервисом.

UART настроен:

raw mode

115200 8N1 (или выбранная скорость)

no flow control

6.2. Порядок действий (алгоритм)

Алгоритм обновления со стороны T113:

Получить firmware по Ethernet:

сохранить файл firmware.bin

проверить размер >0 и <= (FLASH_END-APP_START+1)

вычислить CRC32_total

Открыть UART /dev/ttyS4

Перевести устройство в bootloader без GPIO:

отправить в APP команду ENTER_BOOT (например BOOT\n)

ждать перезагрузки и появления ответа BL:

периодически отправлять SYNC 55 AA 55 AA

ожидать BL>OK или ACK в течение T_BOOT_WAIT = 5 s

если BL не ответил — завершить с ошибкой “device not in bootloader”

Запросить INFO:

CMD_INFO

проверить совпадение APP_START/ограничений

ERASE:

CMD_ERASE(APP_START, image_length_rounded_to_pages)

дождаться ACK (таймаут T_ERASE = 10 s или больше, зависит от размера)

WRITE:

отправлять блоками chunk_size (например 1024 байта):

CMD_WRITE(addr, len, crc32(chunk), chunk_data)

ждать ACK на каждый блок

при NAK: повторить блок до N_RETRY = 3, иначе аварийно завершить

VERIFY:

CMD_VERIFY(APP_START, image_len, CRC32_total)

ожидать ACK

при ошибке: не выполнять RUN; можно повторить WRITE/VERIFY или оставить устройство в BL

RUN:

CMD_RUN

закрыть UART

Логирование:

логировать время, версию BL, результат, коды ошибок.

6.3. Ошибки и поведение T113

При любой ошибке на этапе 3–7:

прекратить RUN

оставить устройство в BL (не отправлять RUN)

сформировать отчёт/лог с error_code

При потере связи в середине WRITE:

повторить синхронизацию и продолжить с последнего подтверждённого блока (если поддерживается), либо начать заново (минимально допустимо — начать заново).

7. Нефункциональные требования
7.1. Надёжность и восстановление

BL должен быть неизменяемым во время update (запрет записи в его область).

При неуспешном update BL должен оставаться рабочим и позволять повторить update.

7.2. Временные ограничения

BL должен отвечать на SYNC не позднее 200 ms после входа в режим обновления.

Таймаут ожидания данных в BL при CMD_WRITE: T_DATA = 2 s (настраиваемо).

7.3. Совместимость сборки

Изменения должны интегрироваться в текущий проект без перехода на Arduino HAL; стиль — CMSIS/регистры.

Должны быть добавлены:

отдельная цель сборки bootloader

линкер-скрипт для BL и APP (или один параметризуемый)

документация по адресам и прошивке

8. Acceptance Criteria (критерии приёмки)

При работающем APP:

команда ENTER_BOOT от T113 приводит к reset и входу в BL без GPIO.

BL отвечает на SYNC и INFO на /dev/ttyS4.

T113 успешно выполняет полный цикл:

ERASE → WRITE → VERIFY → RUN

После RUN:

APP стартует, UART работает, версия APP подтверждается.

При ошибке CRC блока:

BL возвращает NAK с CRC mismatch, повтор WRITE блока возможен.

BL никогда не пишет в область [BL_START .. BL_START+BL_SIZE).

9. Параметры для конфигурации (константы)

MAGIC_UPDATE = 0xB00710AD

GPBR_INDEX = 0

BL_SIZE = 32768

APP_START = 0x00088000

BAUD = 115200 (опционально 921600)

CHUNK_SIZE = 1024

N_RETRY = 3
